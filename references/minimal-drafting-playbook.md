# Minimal Drafting Playbook

Use this playbook to prevent context overload and over-eager drafting.

## Core Rule

Do not read the whole knowledge base by default.

Read only the smallest set of references needed for the current product.

## Default Read Order

1. Read the PRD and user instructions.
2. Read [document-scope-gate.md](document-scope-gate.md).
3. Read [product-intake-schema.md](product-intake-schema.md).
4. Read [template-selection.md](template-selection.md).
5. Read one of the following only if needed:
   - [online-reference-matrix.md](online-reference-matrix.md)
   - [agreement-writing-style.md](agreement-writing-style.md)
   - [output-contract.md](output-contract.md)
6. Read a specific local capture only after an archetype is chosen and only for that archetype.

## What Not to Do

- do not read every file under `references/source-captures/`
- do not read every example file before drafting
- do not use the existence of a polished template to fill missing facts
- do not turn common industry patterns into factual statements
- do not output a long polished privacy policy if the PRD only describes hardware features and gives no real data-processing facts

## Minimal Working Set

For most tasks, the working set should be:

- the PRD itself
- `document-scope-gate.md`
- `product-intake-schema.md`
- `template-selection.md`
- one archetype source or one style guide if necessary

## Escalation Rule

Only expand context when one of the following is true:

- the product type is unclear
- the draft needs a benchmark structure
- the user explicitly asks for richer style alignment
- a high-risk clause needs a verified public source pattern

## Output Discipline

If the fact base is thin, the draft should look conservative.

A shorter, placeholder-heavy draft is better than a longer but fabricated one.
